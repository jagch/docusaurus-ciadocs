Lineamientos para el Diseño de APIs

Versión 1.1 01/06/22

Introducción​

El presente documento contiene las políticas y estándares que se debe seguir para el diseño de APIs en la Capa de Integración de Alicorp (CIA).

Estos lineamientos tienen como objetivo asegurar un diseño de APIs que:

  • Permita que sean fáciles de entender e interpretar.
  • Cuenten con una estructura común y estándar que simplifique la interoperabilidad.
  • Faciliten su uso y composición.
  • Sean consistentes con el resto de APIs.
  • Permita un adecuado gobierno y gestión del ciclo de vida.

Lineamientos Generales​

  • Se debe tomar como base los principios de diseño de servicios publicados en el sharepoint de arquitectura publicados en Sharepoint (Link).
  • Para la implementación de nuevas APIs se deberá seguir el enfoque de API First, en la que diseñamos, documentos y generamos la especificación de las APIs antes de empezar con la construcción.
  • El estándar a utilizar para elaborar las especificaciones es el definido por Open API v3:
    • El archivo de especificación debe publicarse y versionarse en el repositorio definido en Azure DevOps (Link).
    • El idioma a utilizar en las especificaciones es el Inglés.

Metadata​

Las especificación de las API debe considerar como mandatorio los siguientes atributos como metadata:

  • #/info/title : Nombre o identificador único que describa funcionalmente al API.
  • #/info/version : Indica la versión del API, bajo lo definido en el apartado de versionado.
  • #/info/description : Descripción funcional detallada y alcance de la API.
  • #/info/x-audience : Utilizado para indicar el público objetivo del API. Estos pueden ser:
    • internal: Los consumidores de estas APIs son únicamente aplicaciones internas de la compañía.
    • external: Los consumidores de estas APIs son aplicaciones de terceros pero que son partners de Alicorp o soluciones SaaS.
    • public: Los consumidores de estas API puede ser cualquier empresa o usuario final en internet.

URI​

  • El path de la URI estará compuesto por la "versión Mayor" del API, seguido por el namespace, recursos y sub-recursos en caso aplique. Por ejemplo:

    https://rgw.prd.api-alicorp.com/v1/dex/customer/15587846

    Fragmento del PathDescripciónDefinición
    v1Versión mayor del APILa versión mayor de una API se utiliza cuando dos versiones son incompatibles con versiones anteriores de la misma API, como se indica en el punto 8: versionado
    dexNamespaceSe utiliza para proporcionar un contexto y alcance a los recursos. Un Namespace puede referirse a una línea de negocio, capacidad de negocio o a un dominio funcional.
    customerNombre del RecursoUn recurso es un objeto con datos asociados, relaciones con otros recursos y un conjunto de métodos que operan sobre él. Todo aquello que pueda ser nombrado es un recurso, por ejemplo: Person, Image, Document, Shipment, etc.
    No hacer uso de verbos como nombre de un recurso.
    15587846Identificador del RecursoPara obtener un recurso en particular, se debe enviar el identificador del recurso como parte de la URI.

  • Utilizar path normalizados donde se evite finalizar el path con slash o que contenga un doble slash en la URI. Ejemplos de path incorrecto: /products/stock/ o /products//stock.

  • Evitar usar acciones como parte de la URI ya que van en contra del concepto de orientación a recursos de Restful. Sin embargo, puede darse el caso en que el escenario de integración lo amerite, por lo que estará sujeto a evaluación.


Métodos HTTP​

  • GET: El propósito del método GET es leer un recurso o una colección de recursos.

    • Los métodos GET no deben enviar parámetros en el body del request.
    • Solo en casos excepcionales donde se necesita enviar muchos parámetros de entrada para una consulta se puede utilizar el método POST.

  • POST: El principal propósito del método POST es crear un recurso.

    • Si el recurso no existe y es creado, se debe retornar un código 201.
    • En caso el recurso ya exista y sea actualizado, se debería retornar un 200.

  • PUT: El método PUT es utilizado para actualizar un recurso aunque también podría crearlo en caso el recurso no existiese.

    • La ejecución exitosa de este método debería retornar un 204 ya que no se esperaría ningún dato en la respuesta.

  • PATCH: Este método es utilizado para actualizar parte de la data de un recurso.

    • La ejecución exitosa de este método debería retornar un 204 ya que no se esperaría ningún dato en la respuesta.

  • DELETE: Este método es utilizado para eliminar un recurso por su ID.

    • La ejecución exitosa de este método debería retornar un 204 ya que no se esperaría ningún dato en la respuesta.

Payload​

  • El formato de mensaje a utilizar en el payload de las APIs es JSON (encoding UTF-8).
  • En caso se requiera hacer uso de formatos de datos estructurados que no sean JSON (Ejm: XML, CSV), se podrá evaluar como excepción para casos de uso específicos en los que el consumidor tenga una restricción técnica y solo soporte alguno de los formatos mencionados.

Versionado​

Open API permite especificar la versión de una API en el atributo #/info/version. El estándar a utilizar para el versionamiento será el de Semantic Versioning 2.0.0, que propone el formato <Mayor>.<Menor>.<Fix>. Ejemplo: 1.2.11

Consideraciones​

  • Todo cambio realizado a una API productiva debe incrementar su versión.
  • Un versionado Fix se utiliza cuando al introducir un cambio que corrige un bug o compartamiendo indebido del API y que además es compatible con la versión anterior.
  • Un versionado Menor es aquel que introduce nueva funcionalidad o mejora que además es compatible con la versión anterior. (Ejemplo: una nueva operación).
  • Un versionado Mayor es aquel que introduce nueva funcionalidad o cambio pero esta es incompatible con versiones anteriores, lo cual impacta a los consumidores vigentes del API.
  • La versión Mayor forma parte del endpoint, tal como se inidica en la sección URI. Ejemplo: /V1/products/{productId}

Ejemplo:

openapi: 3.0.2 info: title: Stock API description: API para consulta de stock version: 1.1.7

Seguridad​

El mecanismo de autenticación estándar para el consumo de APIS a utilizar es OAuth 2.0. En el caso de la CIA, el oAuth provider es API Connect y el tipo de grant-type a utilizar es Client Credentials.

Por otro lado, a la hora de diseñar una API se debe tener en cuenta los estándares propuestos por OWASP, los cuales han sido recogidos y complementados en el documento de lineamientos de seguridad para el diseño de APIs, elaborado y publicado por el área de Ciberseguridad de Alicorp en el siguiente Link.


Manejo de Errores​

De acuerdo a la especificación de HTTP, el resultado de una ejecución debe retornar un código (status code) y un mensaje, el cual debe poder ser leído e interpretado fácilmente para tener claridad de la respuesta de una invocación. Los status code HTTP 4XX indican que el error se originó del lado del cliente mientras que los status code 5XX que fue del lado del servidor.

Es por ello que se hará uso de una estructura estándar para el manejo de errores que contemple lo anterior y además brinde información adicional para fines de trazabilidad así como el detalle necesario que permita a los consumidores controlar los distintos escenarios de error.

Una respuesta de error debe considerar el esquema error.json, el cual se describe a continuación:

  • code: Nombre único para el error. El nombre describe la clase del error.
  • message: Mensaje de error que sea legible y que describa la clase o tipo de error.
  • details: Contiene un arreglo con los errores específicos encontrados en la petición. Este campo es requerido solo para los errores del lado de cliente (4XX).
    • field: Puntero JSON hacia el campo con el error en caso de encontrarse en el body. Sino, indicar el nombre del parámetro.
    • issue: Código de error de tipo texto que sea legible.
    • description: Una descripción que explique y describa el problema encontrado (issue).

Anexos​

A. error.json​

Esquema JSON estándar para el manejo de los errores en la especificación de las APIs:

error.json

{
    "type": "object",
    "title": "Error",
    "description": "The error details.",
    "properties": {
        "code": {
            "type": "string",
            "description": "The human-readable, unique name of the error."
        },
        "message": {
            "type": "string",
            "description": "The message that describes the error."
        },
        "details": {
            "type": "array",
            "description": "An array of additional details about the error.",
            "items": {
                "$ref": "./error_details.json"
            }
        }
    },
    "required": [
        "name",
        "message",
    ],
}

B. error_details.json​

Esquema JSON estándar para el detalle de los errores de validación de parámetros. Solo aplica para errores del lado de cliente y es complementario al esquema descrito en error.json:

error_details.json

{
    "title": "Error Details",
    "type": "object", 
    "description": "The error details. Required for client-side `4XX` errors.",
    "properties": {
        "field": {
            "type": "string",
            "description": "The field that caused the error. If this field is in the body, set this 
                           value to the field's JSON pointer value. Required for client-side errors."
        },
        "issue": {
            "type": "string",
            "description": "The unique, fine-grained application-level error code."
        },
        "description": {
            "type": "string",
            "description": "The human-readable description for an issue. The description can change over 
                            the lifetime of an API, so clients must not depend on this value."
        }
    },
    "required": [
        "issue"
    ]
}
